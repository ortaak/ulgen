## 🔧 Teknik Detaylar

### Karşılaşılan Sorunlar & Çözümleri

**Sorun 1: `JWT_SESSION_ERROR — JWEDecryptionFailed`**
- **Neden:** Eski bilgisayardan kalan tarayıcı cookie'si, yeni `NEXTAUTH_SECRET` ile uyumsuz
- **Çözüm:** Tarayıcıda localhost:3000 cookie'leri temizlendi (veya incognito pencere kullanıldı)
- **Öğrenilen:** Bilgisayar taşımalarında ilk açılışta cookie temizlemek gerekiyor

**Sorun 2: `Missing token. Please set the UPLOADTHING_TOKEN environment variable`**
- **Neden 1 (API değişikliği):** UploadThing v7'de `UPLOADTHING_SECRET` + `UPLOADTHING_APP_ID` → tek `UPLOADTHING_TOKEN` olarak değişti
- **Neden 2 (Monorepo yapısı):** Token root `/.env`'e eklendi, oysa Next.js `apps/web/.env`'i okuyor
- **Çözüm:** `apps/web/.env`'e `UPLOADTHING_TOKEN` eklendi
- **Öğrenilen:** Bu monorepo'da iki ayrı `.env` var — Next.js sadece kendi dizinindekini okur

**Sorun 3: `EPERM: operation not permitted — webpack cache`**
- **Neden:** Windows Defender / başka bir process webpack cache dosyasını kilitledi
- **Çözüm:** Yok — sadece uyarı, uygulamayı etkilemiyor
- **Öğrenilen:** Windows ortamında sık görülen cosmetic bir uyarı

## 📚 Öğretici Notlar

### 💡 NEDEN İKİ AYRI .env?
━━━━━━━━━━━━━━━━━━━━
**Durum:** Monorepo'da root `.env` ve `apps/web/.env` farklı amaçlara hizmet eder.

**Ne işe yarar:** Next.js kendi `cwd`'sinden `.env` okur — üst dizinleri taramaz.
Prisma da kendi package dizininden okur. Turborepo `globalDependencies` yalnızca
cache invalidation için bu dosyaları izler, içeriklerini workspace'ler arası paylaşmaz.

**Pratik sonuç:** Yeni servis eklendiğinde hem `apps/web/.env` hem de `apps/web/.env.example`
güncellenmelidir. Root `.env.example` genel referans belgesi olarak kullanılabilir.
